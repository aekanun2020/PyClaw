# PDPA Grounding Discipline

กฎเหล็ก: **ห้ามอ้างเลขมาตราหรือเนื้อหากฎหมายจากความจำ** — ทุกมาตราที่ปรากฏในคำตอบต้องมาจากผล tool call ในรอบเดียวกันนี้เท่านั้น. ถ้าไม่ได้ retrieve มา ห้ามเขียนเลขมาตรานั้น.

## Gotcha (อ่านก่อนทุกครั้ง)
จากการทดสอบจริง พบรูปแบบความผิดพลาดซ้ำๆ ที่ skill นี้มีไว้เพื่ออุดโดยตรง:
- **อ้างมาตราจากความจำโดยไม่ retrieve** เช่น ม.23 ที่มักถูกอ้างเองทุกครั้งที่พูดเรื่อง "แจ้งวัตถุประสงค์" ทั้งที่ไม่มี tool call ใดดึงมา — บางครั้งเนื้อหาบังเอิญถูก แต่ก็ยังผิดวินัย grounding.
- **เลขมาตรา hallucination** เช่นอ้าง ม.29 เรื่อง security measures ทั้งที่ ม.29 จริงเป็นเรื่อง cross-border transfer.
- **เห็นแต่ไม่ใช้** retrieve เจอ relation ที่ตรงกว่า (เช่น `sec_27 → REFERENCES_SECTION → sec_39`) แต่กลับไปอ้างมาตราอื่น (ม.21) ที่ขัดกับสิ่งที่ต้องการสื่อ.
- **สับสนฐานเก็บกับฐานใช้/เปิดเผย** เคยตอบกรณีกล้องวงจรปิด โดยอ้าง ม.24(5) (legitimate interest) เป็นฐานทั้งการ *ติดตั้ง/เก็บภาพ*, การ *สอบสวนพนักงาน*, และการ *โพสต์การตลาด* — ผิด. ม.24(5) เป็นฐาน **เก็บ** เท่านั้น ส่วนการนำภาพไป **ใช้/เปิดเผย** (สอบสวน vs การตลาด) ต้องวินิจฉัยภายใต้ **ม.27** คนละมาตรา คนละขั้นตอน.
- **อ้าง ม.21 เป็นฐานทางกฎหมาย** ม.21 เป็น **หลักการ (purpose limitation principle)** ไม่ใช่ lawful basis. การ "ใช้นอกวัตถุประสงค์" ผูกกับ **ม.27** (ฐานควบคุมการใช้/เปิดเผย) ไม่ใช่ ม.21.
- **แทรกศัพท์ข้ามเขตอำนาจ** เช่น SCCs / BCRs (เป็นศัพท์ GDPR ของ EU) มาใส่ในคำตอบ PDPA ไทย ทั้งที่ไม่มีในตัวบทและไม่มี grounding.
- **บั๊กภาษาปน** เคยมีตัวอักษรภาษาอื่น (เช่นจีน) หลุดในคำตอบไทย — ต้องตรวจก่อนส่ง.

## Workflow (บังคับ)
1. **แยกขั้นตอนก่อน — เก็บ หรือ ใช้/เปิดเผย?** ก่อนอะไรทั้งหมด ถามตัวเองว่าคำถามกำลังถามถึงขั้นไหน — (ก) การ **เก็บรวบรวม** → ฐานอยู่ที่ **ม.24** (ข้อมูลทั่วไป) หรือ **ม.26** (ข้อมูลอ่อนไหว); (ข) การ **ใช้หรือเปิดเผย** (รวมถึง สอบสวน/ส่งต่อ/โพสต์/ทำตลาด) → ฐานอยู่ที่ **ม.27** — **ห้ามใช้ ม.24/26 ตอบคำถามเรื่องการใช้/เปิดเผย โดยตรง** — ดูหัวข้อ "Collect vs Use/Disclose Gate" ด้านล่าง.
2. **แยกประเภทคำถามการใช้/เปิดเผย:** เป็นการใช้ "ตามวัตถุประสงค์เดิม (compatible purpose)", "นอกวัตถุประสงค์เดิม (new purpose)", หรือ "วัตถุประสงค์ภายในอื่นที่ได้รับยกเว้น consent" (เช่น สอบสวนพนักงาน) — การตั้งคำถามผิดประเภทตั้งแต่ต้นทำให้คำตอบผิดทั้งสาย ต่อให้ค้นแม่นแค่ไหน. compatible purpose → ใช้ได้ตาม ม.27 + บันทึก ROPA; new purpose → ต้องขอ consent ใหม่.
3. **ค้นก่อนอ้างเสมอ:** เรียก `pdpa_search_pdpa` ก่อนระบุมาตราใดๆ. ห้ามเขียนเลขมาตราก่อนเห็นผล tool.
4. **Expand query ด้วยเลขมาตรา + ศัพท์ตัวบท:** semantic search จับคู่ขึ้นกับคำ query มาก — ถ้าค้นด้วยภาษาเชิงสถานการณ์อย่างเดียว (เช่น "สอบสวนพนักงาน") มักดึงมาตราเป้าหมายไม่ติด. ให้ค้นเพิ่มด้วยเลขมาตราตรงๆ และศัพท์ในตัวบท (เช่น `มาตรา 27 ใช้เปิดเผยข้อมูล`, `มาตรา 27 วรรคสาม วัตถุประสงค์`) เมื่อสงสัยว่ามีมาตราเฉพาะที่ตรงกว่า. **ถ้าคำถามเกี่ยวกับการใช้/เปิดเผย ต้องค้น ม.27 เสมอ** (เช่น `มาตรา 27 ห้ามใช้หรือเปิดเผย ความยินยอม`) และถ้าพูดถึงการบันทึกต้องค้น ROPA / ม.39 ด้วย.
5. **ตรวจ related sections:** เรียก `pdpa_get_related_sections` แล้ว**ใช้ relation ที่เจอจริง** — ถ้าผลชี้ไปมาตราที่ตรงประเด็นกว่า (เช่น `sec_27→sec_39`) ต้องเลือกใช้มาตรานั้น ห้ามมองข้ามไปอ้างมาตราอื่นจากความจำ.
6. **ก่อนระบุตัวเลขโทษปรับ:** เรียก `pdpa_get_penalty` ทุกครั้ง — ห้ามเขียนจำนวนเงินหรือโทษจากความจำ. ระบุให้ชัดว่าเป็นโทษ controller หรือ processor และเข้าเงื่อนไขมาตราใด. **ระวังโทษตามประเภทข้อมูล:** ฐานข้อมูลทั่วไปกับข้อมูลอ่อนไหว (ม.26) มีโทษต่างระดับกัน — อย่าเหมารวมขึ้นกับ `pdpa_get_penalty` ที่คืนมา.
7. **ถ้า Graph ไม่คืนข้อมูล:** ระบุตรงๆ ว่า "ไม่มีข้อมูลใน Graph" แล้วตอบเท่าที่ retrieve ได้ — ห้ามเติมมาตราจากความจำมาอุดช่องว่าง.

## Citation-strict (มิติสำคัญที่สุด)
ก่อนเขียนเลขมาตราทุกครั้ง ถามตัวเองว่า: **"เลขมาตรานี้ปรากฏในผล tool call ของรอบนี้หรือยัง?"**
- ถ้า **ใช่** → อ้างได้ พร้อมยึดเนื้อหาตามที่ tool คืนมา ไม่ใช่ตามที่จำได้.
- ถ้า **ไม่** → มี 2 ทางเลือกเท่านั้น: (ก) ค้นเพิ่มด้วย `pdpa_search_pdpa` ให้ retrieve มาก่อน หรือ (ข) ถ้ายังหาไม่เจอแต่จำเป็นต้องพูด ให้ติด `[Agent inference]` กำกับให้ชัด และห้ามฟันธงเด็ดขาด.

## Collect vs Use/Disclose Gate (ม.27) — หัวใจของ skill นี้
**หลักเหล็ก:** ม.24/26 = ฐานการ **"เก็บ"** (collect). ม.27 = ประตูควบคุมการ **"ใช้/เปิดเผย"** (use/disclose). มีฐานเก็บไม่ได้แปลว่าใช้/เปิดเผยได้เสมอ — ต้องผ่าน "ประตูสองชั้น".

ถ้อยคำถามพูดถึงการ นำข้อมูลไป **ใช้ / เปิดเผย / ส่งต่อ / สอบสวน / โพสต์ / ทำตลาด / แชร์** → นี่คือคำถาม **ม.27** — ห้ามตอบด้วย ม.24(5) ลอยๆ เหมือนเป็นคำตอบสำเร็จรูป.

**Decision tree (บังคับเดินตาม หลัง retrieve มาตราจริงแล้ว):**
```
Q1. ข้อมูลนี้ถูก "เก็บ" มาโดยฐานอะไร?
   - consent           → การใช้/เปิดเผยต้องขอ consent สำหรับวัตถุประสงค์นั้นด้วย
   - ฐานยกเว้น ม.24 / ม.26  → ไป Q2

Q2. การใช้/เปิดเผยนี้อยู่ในขอบวัตถุประสงค์ที่แจ้งไว้หรือไม่? (ม.27 ประกอบ ม.21)
   - อยู่ในขอบ (compatible) → ใช้/เปิดเผยได้ตาม ม.27 วรรคหนึ่ง + **บันทึก ROPA ตาม ม.27 วรรคสาม → ม.39**
   - นอกขอบ (new purpose) → ต้องขอ consent ใหม่ (ม.21 + ม.27)

Q3. ข้อมูลที่ใช้/เปิดเผยเป็นข้อมูลอ่อนไหว (ม.26) หรือไม่?
   - ใช่ → ความเสี่ยงโทษหนักขึ้น (อาจมีโทษอาญา + ปกครองระดับ sensitive) — ยืนยันด้วย pdpa_get_penalty
   - ไม่ใช่ → โทษระดับข้อมูลทั่วไป — ยืนยันด้วย pdpa_get_penalty
```

**ตัวอย่าง CCTV (เคสที่เคยตอบผิด):**
- ติดกล้องเพื่อความปลอดภัย = ฐาน **เก็บ** ม.24(5) (legitimate interest)
- นำภาพไป **สอบสวนพนักงาน** ที่ต้องสงสัย = การ **ใช้** → วินิจฉัยภายใต้ **ม.27** (compatible purpose กับความปลอดภัย) + บันทึก ROPA ตาม ม.27 วรรคสาม → ม.39 — **ไม่ใช่** การอ้าง ม.24(5) ซ้ำ
- นำภาพไป **โพสต์โซเชียล/การตลาด** = **new purpose** → ผิด ม.27 ต้องขอ consent ใหม่
- **ห้ามอ้าง ม.21 เป็นฐานที่ทำให้การตลาดผิด** — ม.21 เป็นหลักการ ตัวฐานควบคุมจริงคือ ม.27

ทุกมาตราข้างต้น (ม.24/26/27/39) **ต้อง retrieve มาจริงก่อนอ้าง** — decision tree นี้บอกว่าต้องไปหามาตราไหน ไม่ได้อนุญาตให้อ้างเลขมาตราจากความจำ.

## ห้ามแทรกศัพท์ข้ามเขตอำนาจ (No cross-jurisdiction terms)
PDPA ไทยกับ GDPR ของ EU เป็นคนละกฎหมาย. ห้ามนำศัพท์/กลไกของ GDPR มาใส่ในคำตอบ PDPA เว้นแต่ tool result คืนคำนั้นมาจริง:
- ห้ามใช้ **SCCs (Standard Contractual Clauses)**, **BCRs (Binding Corporate Rules)**, "adequacy decision" แบบ EU ฯลฯ เป็นข้อกำหนดของ PDPA.
- เรื่อง cross-border transfer ให้ยึดถ้อยคำของตัวบท PDPA ที่ retrieve มาจริง (เช่น "มาตรการคุ้มครองที่เหมาะสม", "บังคับตามสิทธิของเจ้าของข้อมูลได้", "มาตรการเยียวยาทางกฎหมายที่มีประสิทธิภาพ" ตาม ม.28) ไม่ใช่ศัพท์ GDPR.

## Calibration (ความมั่นใจต้องสอดคล้องกับ grounding)
- คำตอบที่มี grounding ครบ → ตอบหนักแน่นได้.
- คำตอบที่ retrieve ไม่เจอ → **ต้องลดความมั่นใจลงตามจริง** อย่าฟันธงด้วยน้ำเสียงเดียวกับคำตอบที่มีหลักฐาน. ผู้อ่านต้องแยกออกว่าส่วนไหน grounded ส่วนไหนเป็น inference.
- เมื่อไม่แน่ใจในประเด็นที่มีความเสี่ยงทางกฎหมาย ให้เลือกตีความเข้มงวด/ปลอดภัยไว้ก่อน (เช่น แนะนำให้ขอ consent เพิ่ม) แทนการฟันธงว่า "ทำได้เลย".

## Validation (ก่อนส่งคำตอบ — เช็คทุกข้อ)
1. ทุกเลขมาตราในคำตอบ → ปรากฏในผล tool call ของรอบนี้ (ถ้าไม่ → ลบหรือติด [Agent inference]).
2. เนื้อหาของแต่ละมาตรา → ตรงกับที่ tool คืนมา ไม่ใช่ตามความจำ.
3. ไม่มีศัพท์ GDPR (SCCs/BCRs/adequacy แบบ EU) ที่ไม่ได้มาจาก tool.
4. ไม่มีตัวอักษรภาษาอื่นปนในคำตอบภาษาไทย.
5. คำตอบในแต่ละส่วน **ไม่ขัดแย้งกันเอง** (เช่น หัวข้อบอกใช้ ม.24(5) แต่ตารางสรุปบอก ม.21) — ถ้าขัดกัน แก้ให้สอดคล้องก่อนส่ง.
6. **แยกฐานเก็บ/ใช้ถูกต้อง:** ถ้าคำถามถามถึงการ ใช้/เปิดเผย/สอบสวน/โพสต์ → คำตอบต้องอ้าง **ม.27** เป็นฐานควบคุม (ไม่ใช่ ม.24/26 ลอยๆ และไม่ใช่ ม.21). ถ้าเป็น compatible purpose → ต้องมี ROPA (ม.27 วรรคสาม → ม.39) ระบุไว้ในคำตอบ.
7. ตัวเลขโทษทุกตัว → มาจาก `pdpa_get_penalty` และระบุ controller/processor ถูกต้อง.

## Recovery (เมื่อ retrieve ไม่เจอมาตราที่ควรมี)
ถ้าค้นด้วยภาษาเชิงสถานการณ์แล้วไม่เจอมาตราที่น่าจะตรง (อาการนี้พบบ่อยกับ ม.21, 23, 27):
1. ค้นซ้ำด้วย **เลขมาตราตรงๆ + ศัพท์ในตัวบท** (เช่น `มาตรา 27 ใช้หรือเปิดเผยข้อมูล วัตถุประสงค์`) — เนื้อหามักมีอยู่ใน Graph แต่ดึงติดเฉพาะเมื่อ query ตรงคำ.
2. ใช้ `pdpa_get_related_sections` จากมาตราที่เกี่ยวข้องที่ดึงได้แล้ว เพื่อไล่ไปยังมาตราเป้าหมายผ่าน relation.
3. ถ้ายังไม่เจอจริง → ระบุ "ไม่มีข้อมูลใน Graph สำหรับประเด็นนี้" และห้ามเดาเลขมาตรามาแทน.
4. ห้ามเรียก tool เดิมด้วย query เดิมซ้ำ — ต้องเปลี่ยนคำค้นทุกครั้ง.
